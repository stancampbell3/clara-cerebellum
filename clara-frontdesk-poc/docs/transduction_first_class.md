# Transduction First Class

**Branch:** `housbonde_lif`
**Date:** 2026-05-03
**Status:** Planning — iterate before implementing
**Follows:** `stoke_the_coals.md`

---

## Background

The Clara architecture has a core claim: Prolog rules (backward chaining) and CLIPS
rules (forward chaining) work together in the reasoning cycle, and the CLIPS rules are
*derived automatically* from the Prolog source via the Clara transducer. Showing this
derivation live is part of the POC's purpose.

During the Stoke the Coals implementation we subverted this in two ways:

1. **`front_desk_poc_reprise_clara.pl`** — marked "auto-generated" in its header but
   edited directly. The transducer should produce this file from the source
   `front_desk_poc_reprise.pl`; we maintained it by hand.

2. **`front_desk_poc_reprise_clara.clp`** — should be regenerated by running
   `transduction --decorate` after any Prolog rule change. Instead we manually wrote
   the new `redirect/3` CLIPS rules (30–31), including manually choosing to continue
   the existing numbering sequence.

This happened because the transducer step has friction: it is documented but not
enforced, so it was skipped. The result is that the generated files are now hand-patched
artifacts, not clean transducer output. Beyond the immediate mess, the experience
exposed several structural gaps in how transduction fits into the development workflow.

---

## Issues

### Issue 1 — No automation enforces regeneration

The transduction step is documented ("run `transduction --decorate`") but nothing
enforces it. There is no `make transduct`, no pre-commit hook, no CI check. The gap
between editing the source `.pl` and running the transducer is purely manual discipline.
Discipline fails under time pressure; we skipped the step twice in a row.

**Effect:** Generated files drift silently from their source. The demo shows generated
output, not freshly transduced output, and there is no visible indicator of staleness.

---

### Issue 2 — Generated files live alongside source with no clear boundary

`front_desk_poc_reprise.pl` (source) and `front_desk_poc_reprise_clara.pl` (generated)
sit side by side in `roost/`. They look like peer files. Nothing in the file system
layout, the git history, or the file headers makes it obvious that one should never be
edited and must always be reproduced from the other.

**Effect:** Any contributor — or future Claude session — will treat them as peers and
edit whichever is convenient, which is what happened here.

---

### Issue 3 — Output-variable positions in CLIPS goals are unresolved

Existing CLIPS forward-chaining rules publish fully-grounded Prolog goals where every
argument is a known atom:

```clp
(coire-publish-goal (str-cat "suggestion(" ?Visitor ",'Greet the visitor.')" ))
```

This works because the suggestion text is fixed at transduction time. For `redirect/3`
the first clause has fixed second and third arguments (`'This visitor appears lost or
confused…'`, `'nearest map kiosk'`), so grounding is also possible. But the general
case — predicates where output positions carry variables that should be unified during
evaluation — is unsolved.

We wrote:
```clp
(coire-publish-goal (str-cat "redirect(" ?Visitor ",_Reason,_Where)"))
```

`_Reason` and `_Where` are valid Prolog anonymous variable names in the atom string, so
the goal will unify against any matching `redirect/3` clause. But whether those bindings
are returned through the `coire-publish-goal` pathway and visible to the next cycle is
untested. The transducer has no explicit handling for this case — it was not encountered
before `redirect/3` introduced output-variable positions in a decision predicate.

**Effect:** CLIPS forward chaining for predicates with unbound output variables is
undefined behaviour. The correctness of rules 30–31 is unknown until tested.

---

### Issue 4 — Transducer is role-blind: all multi-clause predicates are treated the same

The transducer generates the same forward-chaining pattern for `suggestion/2`,
`admit/2`, and `redirect/3`. But these predicates have different roles:

| Predicate | Role | When re-evaluation matters |
|---|---|---|
| `suggestion/2` | Pure query; result is guidance text for the LLM | Any time a relevant fact changes |
| `admit/2` | Decision predicate; first match wins via `->` | Any time a relevant fact changes — but result should trigger a full `daemonic_turn` re-run |
| `redirect/3` | Decision predicate; checked after all `admit/2` rules fail | Same as `admit/2` |

A role-blind transducer generates CLIPS rules that push isolated `admit` or `redirect`
goals back to Prolog. But what the cycle actually needs is for a change to an
`admit/2`-relevant fact to trigger a fresh `daemonic_turn/5` evaluation, not a
standalone `admit(visitor,_Reason)` probe. The current design conflates "I need to
re-check this predicate" with "I need to re-run the decision logic that uses it."

**Effect:** The CLIPS forward-chaining rules for decision predicates may fire correctly
at the Prolog level but produce no useful cycle-level effect unless the cycle controller
knows to treat a successful `admit` or `redirect` sub-goal as a trigger to re-run the
root goal.

---

### Issue 5 — Rule numbering is fragile and manual

CLIPS requires unique rule names. The transducer appends sequential integers
(`transduced-suggestion-on-visitor-14`, `…-15`, etc.). When we add rules by hand we
must know the current high-water mark (29) and continue from there (30, 31). This is
error-prone and unscalable — two contributors editing in parallel would collide.

Content-based names (e.g., `transduced-redirect-on-lost_or_confused`) would be
self-documenting, collision-free, and not require tracking a counter. The transducer
should generate names this way; sequential integers are an implementation shortcut that
leaks into every generated artifact.

## Suggested Paths

### Path A — Enforce regeneration with a build target

Add a `Makefile` (or `justfile`) target at the workspace root:

```make
transduct:
    cd clara-frontdesk-poc && \
      transduction --decorate roost/front_desk_poc_reprise.pl \
        -o roost/front_desk_poc_reprise_clara.pl \
        --clips roost/front_desk_poc_reprise_clara.clp
```

With a companion `check-transduct` target that fails if either output file is older than
the source `.pl` (usable in CI and as a pre-commit hook).

This does not require changing the transducer itself. It just makes the workflow
reproducible and enforceable.

**Effort:** Low. **Risk:** Low. Prerequisite for everything else.

---

### Path B — Separate source from generated output

Move generated files out of `roost/` into `roost/gen/`:

```
roost/
  front_desk_poc_reprise.pl          ← source (edit here)
  gen/
    front_desk_poc_reprise_clara.pl  ← generated (do not edit)
    front_desk_poc_reprise_clara.clp ← generated (do not edit)
```

Add `roost/gen/` to `.gitignore` (generated on demand, not committed), OR keep it
committed with a prominent banner at the top of each generated file:

```
% !! AUTO-GENERATED — do not edit. Regenerate with: make transduct !!
```

Update `localnet_dis.toml` and `city_of_dis.toml` `clara_pl_path` / `clara_clp_path`
to point into `gen/`.

**Effort:** Low. **Risk:** Low (path change only). Addresses Issue 2.

---

### Path C — Resolve output-variable handling in the transducer

Define and implement a convention for how the transducer handles predicates with output
variables in their CLIPS goal strings. Two options:

**Option C1 — Ground from rule head when possible.**  
For `redirect(Visitor, 'Fixed reason atom.', 'fixed destination')`, the transducer
grounds the goal string at transduction time:
```clp
(coire-publish-goal (str-cat "redirect(" ?Visitor
    ",'This visitor appears lost or confused and cannot proceed.','nearest map kiosk')"))
```
This is safe for single-clause predicates where the output is fixed. It fails for
multi-clause predicates where each clause has a different output.

**Option C2 — Use named Prolog variables for output positions.**  
The transducer substitutes generated variable names for output positions:
```clp
(coire-publish-goal (str-cat "redirect(" ?Visitor ",Reason,Where)"))
```
`Reason` and `Where` (uppercase, no underscore) are proper Prolog variables. Whether
their bindings need to propagate back to CLIPS depends on how the cycle uses the result.
For decision predicates the bindings are consumed by `daemonic_turn/5` in the Prolog
pass, so the CLIPS rule only needs to signal "re-evaluate redirect" — bindings don't
need to return to CLIPS.

**Recommended:** Option C1 for single-clause predicates with fixed outputs (current
`redirect/3` case). Option C2 for multi-clause or variable-output predicates, with a
clear note that bindings do not return to CLIPS.

**Effort:** Medium. Requires transducer changes. Addresses Issue 3.

---

### Path D — Role annotation in Prolog source

Allow the transducer to distinguish predicate roles via source annotations:

```prolog
%% @role decision
redirect(Visitor, 'This visitor appears lost or confused and cannot proceed.', 'nearest map kiosk') :- ...

%% @role suggestion
suggestion(Visitor, 'Greet the visitor.') :- ...

%% @role entry_point
daemonic_turn(Visitor, Suggestions, Decision, Reason, Where) :- ...
```

The transducer then generates different CLIPS behaviour per role:

- **`suggestion`**: generate CLIPS rules that push the specific suggestion goal (current
  behaviour).
- **`decision`**: generate CLIPS rules that push the *entry point goal*
  (`daemonic_turn(visitor,_,_,_,_)`) rather than the decision predicate in isolation,
  so that a change in any decision-relevant fact triggers a full turn re-evaluation.
- **`entry_point`**: no CLIPS rules generated; this is the Prolog-side root.

This makes the forward-chaining semantics match the intended cycle behaviour.

**Effort:** High (transducer + annotation convention). Addresses Issue 4. Best done
after Paths A–C are stable.

---

### Path E — Content-based CLIPS rule names

Change the transducer to generate CLIPS rule names from predicate name + triggering
fact name rather than sequential integers:

```
transduced-suggestion-greet-visitor-on-visitor
transduced-suggestion-greet-visitor-on-not_greeted
transduced-redirect-lost_confused-on-visitor
transduced-redirect-lost_confused-on-lost_or_confused
```

The naming scheme: `transduced-{predicate}-{rule_summary}-on-{triggering_fact}`.
Collision-free because `(predicate, rule_summary, triggering_fact)` is unique per rule.

**Effort:** Low-Medium (transducer naming change only). Addresses Issue 5. Can be done
alongside Path C.

## Priority Order

| Priority | Path | Why |
|---|---|---|
| 1 | **A** — Build target | Restores the demo narrative; all other improvements depend on the transducer being runnable |
| 2 | **B** — Separate gen/ | Prevents the hand-edit trap from recurring; low effort, high clarity |
| 3 | **C** — Output variables | Fixes the correctness question on rules 30–31; needed before relying on redirect forward chaining |
| 4 | **E** — Content-based names | Low effort cleanup; do alongside C |
| 5 | **D** — Role annotations | Most impactful for cycle correctness but highest effort; plan after A–C–E land |

## Demo Narrative Target

The goal state for the POC demo:

1. Presenter edits a rule in `front_desk_poc_reprise.pl`
2. Runs `make transduct` — terminal shows the transducer producing `gen/_clara.pl` and
   `gen/_clara.clp` live
3. The generated CLIPS rules are visible and explainable: "this CLIPS rule fires
   whenever `lost_or_confused` is asserted, pushing a re-evaluation of the redirect
   decision back into Prolog"
4. The demo runs end-to-end with freshly transduced files, not hand-patched ones

This is the concrete test for whether the transduction step is truly first class.
