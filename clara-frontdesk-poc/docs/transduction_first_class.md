# Transduction First Class

**Branch:** `housbonde_lif`
**Date:** 2026-05-03
**Status:** Planning — iterate before implementing
**Follows:** `stoke_the_coals.md`

---

## Background

The Clara architecture has a core claim: Prolog rules (backward chaining) and CLIPS
rules (forward chaining) work together in the reasoning cycle, and the CLIPS rules are
*derived automatically* from the Prolog source via the Clara transducer. Showing this
derivation live is part of the POC's purpose.

During the Stoke the Coals implementation we subverted this in two ways:

1. **`front_desk_poc_reprise_clara.pl`** — marked "auto-generated" in its header but
   edited directly. The transducer should produce this file from the source
   `front_desk_poc_reprise.pl`; we maintained it by hand.

2. **`front_desk_poc_reprise_clara.clp`** — should be regenerated by running
   `transduction --decorate` after any Prolog rule change. Instead we manually wrote
   the new `redirect/3` CLIPS rules (30–31), including manually picking up the
   numbering sequence from where the previous rules ended.

This happened because the transducer step has friction: it is documented but not
enforced, so it was skipped. The result is that the generated files are now hand-patched
artifacts, not clean transducer output. Beyond the immediate mess, the experience
exposed three structural gaps addressed by this document, and one deeper architectural
question deferred to a separate document.

---

## Issues

### Issue 1 — No automation enforces regeneration

The transduction step is documented ("run `transduction --decorate`") but nothing
enforces it. There is no `make transduct`, no pre-commit hook, no CI check. The gap
between editing the source `.pl` and running the transducer is purely manual discipline.
Discipline fails under time pressure; we skipped the step twice in a row.

**Effect:** Generated files drift silently from their source. The demo shows stale
hand-patched output, not freshly transduced output.

**Addressed by:** Path A.

---

### Issue 2 — Generated files live alongside source with no clear boundary

`front_desk_poc_reprise.pl` (source) and `front_desk_poc_reprise_clara.pl` (generated)
sit side by side in `roost/`. They look like peer files. Nothing in the file system
layout or the file headers makes it obvious that one should never be edited and must
always be reproduced from the other.

**Effect:** Any contributor — or future Claude session — will treat them as peers and
edit whichever is convenient, which is exactly what happened.

**Addressed by:** Path B (choice point — see below).

---

### Issue 3 — Transducer output has two correctness gaps

**3a — Output-variable positions in CLIPS goals.**
Existing CLIPS rules publish fully-grounded goals where every argument is a known atom:

```clp
(coire-publish-goal (str-cat "suggestion(" ?Visitor ",'Greet the visitor.')" ))
```

This works because the suggestion text is fixed at transduction time. For `redirect/3`
the output positions (Reason, Where) are not fixed across all clauses. We hand-wrote:

```clp
(coire-publish-goal (str-cat "redirect(" ?Visitor ",_Reason,_Where)"))
```

`_Reason` and `_Where` are valid Prolog anonymous variable names, so the goal unifies
against any matching clause — but whether bindings propagate back through
`coire-publish-goal` is untested. The transducer has no handling for this case.

**3b — Sequential rule numbering.**
CLIPS requires unique rule names. The transducer appends sequential integers
(`transduced-suggestion-on-visitor-14`, `…-15`, etc.). When adding rules by hand the
writer must know the current high-water mark and continue from it. Two contributors
editing in parallel would produce collisions. Content-based names
(`transduced-redirect-on-lost_or_confused`) would be self-documenting and
collision-free. This is a two-line change in the transducer alongside the output
variable fix, so it is bundled here rather than treated as a separate path.

**Effect:** CLIPS forward chaining for predicates with unbound output variables is
undefined behaviour. The correctness of rules 30–31 is unknown until tested. Rule
numbering will collide under parallel development.

**Addressed by:** Path C (choice point — see below).

## Paths

### Path A — Build target (action: do this)

Add a `Makefile` target in `clara-frontdesk-poc/`:

```make
transduct:
	transduction --decorate roost/front_desk_poc_reprise.pl \
	  -o roost/gen/front_desk_poc_reprise_clara.pl \
	  --clips roost/gen/front_desk_poc_reprise_clara.clp

check-transduct:
	@if [ roost/front_desk_poc_reprise.pl -nt roost/gen/front_desk_poc_reprise_clara.pl ]; \
	  then echo "Generated files are stale — run make transduct"; exit 1; fi
```

`check-transduct` can be wired to CI and to a pre-commit hook so staleness is caught
before it becomes a demo problem.

This does not require changing the transducer itself. **Prerequisite for everything
else.**

---

### Path B — Generated file boundary (choice point)

Three options. Pick one.

**B1 — Banner only.**
Add a prominent header to each generated file:

```prolog
% !! AUTO-GENERATED — do not edit directly.
% !! Source: roost/front_desk_poc_reprise.pl
% !! Regenerate with: make transduct
```

Files stay in `roost/`. TOML configs and Dockerfile are unchanged. Lowest friction.
Does not change the directory layout, so the files still *look* like source peers even
with the banner.

**B2 — `gen/` directory, files committed.**
Move generated files to `roost/gen/`. Update `clara_pl_path` and `clara_clp_path` in
both TOML configs and any Dockerfile `COPY` directives. Committed to the repo — the
repo is always runnable without needing the transducer installed.

The directory boundary makes the source/generated distinction structural and visible. A
contributor opening `roost/` sees one `.pl` file; the generated output is obviously
elsewhere.

**B3 — `gen/` directory, gitignored.**
Same as B2 but `roost/gen/` is in `.gitignore`. Generated files are always built at
build or deploy time. The Dockerfile runs `make transduct` as a build step. The repo
holds only source.

This is the purest form — generated files genuinely never live in source control — but
it requires the transducer to be available wherever the image is built, and breaks
`go.sh` / local dev runs that skip Docker unless those also run `make transduct` first.

**Recommendation: B2.** The directory boundary makes the separation structural without
requiring the transducer at every environment. B3 is architecturally ideal but adds
operational risk for the POC phase.

### Path C — Transducer output correctness (choice point within)

Two sub-problems; fix together in one transducer change.

**C-naming: content-based rule names.**
Replace sequential integer suffixes with a content-based scheme:

```
transduced-{predicate}-{clause-summary}-on-{triggering-fact}
```

Example: `transduced-redirect-lost-confused-on-lost_or_confused`.
Collision-free, self-documenting, no counter to track. This is the simpler of the two
sub-problems and should be done regardless of the output-variable choice.

**C-vars: output-variable positions — choose one.**

**Option C1 — Ground from the rule head (recommended for current codebase).**
When all output arguments of a clause are fixed atoms in the head, the transducer
grounds them into the CLIPS goal string at transduction time:

```clp
(coire-publish-goal (str-cat "redirect(" ?Visitor
    ",'This visitor appears lost or confused and cannot proceed.','nearest map kiosk')"))
```

Works for the current `redirect/3` and any single-clause-per-output predicate.
Fails if a predicate has multiple clauses with different output atoms — the transducer
would need to generate one CLIPS rule per clause, each grounded to that clause's head.
This is feasible and may be the right long-term approach.

**Option C2 — Named Prolog variables for output positions.**
The transducer emits uppercase variable names for unbound output positions:

```clp
(coire-publish-goal (str-cat "redirect(" ?Visitor ",Reason,Where)"))
```

`Reason` and `Where` (uppercase) are proper unbound Prolog variables in the goal
string. The goal succeeds and binds them, but since the CLIPS rule's purpose is to
*signal* that redirect should be re-evaluated (not to consume the bindings in CLIPS),
the bindings are not needed back in CLIPS. This approach is uniform regardless of how
many clauses a predicate has or what its output values are.

The risk: it is not confirmed that bindings from `coire-publish-goal` goals are
discarded cleanly rather than causing errors. This needs a test before committing to C2.

**Recommendation: C1 for now, with multi-clause grounding as a follow-on.**
C1 is safe and covers the current codebase. C2 is more general but carries an untested
assumption about the coire pathway. Test C2 in isolation before adopting it broadly.

## Deferred — Role-aware transduction

The transducer currently treats `suggestion/2`, `admit/2`, and `redirect/3` the same:
it generates CLIPS rules that push isolated predicate goals back to Prolog. But
semantically, `admit/2` and `redirect/3` are decision predicates — when one of their
preconditions changes, the right response is to re-run `daemonic_turn/5` (the root
goal), not to probe `admit` or `redirect` in isolation.

This is a real architectural gap, but it is a different document. It requires:
- A convention for annotating predicate roles in the Prolog source
- Transducer changes to generate role-appropriate CLIPS rules
- Possibly cycle controller changes to treat certain coire events as root-goal triggers

This is tracked separately as **`transduction_role_aware.md`** (to be created when
Paths A–C are stable).

## Demo Narrative Target

The goal state that tests whether transduction is truly first class:

1. Presenter edits a rule in `roost/front_desk_poc_reprise.pl`
2. Runs `make transduct` — terminal shows the transducer producing `roost/gen/_clara.pl`
   and `roost/gen/_clara.clp` live
3. The generated CLIPS rules are visible and explainable: "this CLIPS rule fires
   whenever `lost_or_confused` is asserted, pushing a re-evaluation of the redirect
   decision back into Prolog"
4. The demo runs end-to-end with freshly transduced files, not hand-patched ones
