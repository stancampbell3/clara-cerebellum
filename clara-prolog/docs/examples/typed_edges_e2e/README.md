# Typed Edges — Live E2E Example (auto-pipe)

The canonical two-node Ritual used to verify typed edges: a deduce-capable
Clara node connected to a Groq evaluator by an `offering` edge. As of the
auto-pipe iteration the edge is `pipeMode: "auto"` — the edge itself
forwards the Offering and delivers the reply as a fact; the authored source
never calls a consult helper.

(The original synchronous variant — authored `consult_clara_groq/2` call,
`pipeMode` absent — was verified live 2026-07-05; see the git history of
this directory and `docs/ritual_typed_edges.md`.)

## Files

- `reasoned_response.pl` — the authored Prolog for node `n1`. It only
  *consumes* `edge_result(e1, hohi, GroqReply)` (clause 1, synthesis) or
  `edge_result(e1, tabu, _)` (clause 2, Clara-only fallback). The pipe and
  dispatch rules are **generated by edge transduction** from edge `e1`.
- `graph_layout.json` — the Cobbler graph: `n1` Clara
  (`clara_mind_splinter`, ollama), `n2` Clara/Groq (`groq_evaluator`), one
  `offering` edge `e1` with `"pipeMode": "auto"`. `prologSource` is left
  empty here; `run_e2e.sh` injects `reasoned_response.pl` so the .pl file
  stays the single source of truth.
- `run_e2e.sh` — registers a scratch user, focuses `clara_mind_splinter`,
  creates + activates the config on lildaemon (:6666), and runs a query.

## What happens at runtime

1. Run passes the query to the deduction as a synthetic incoming Offering
   (`initial_offering`, fresh correlation id, topic `run`) along with
   `self_node_id: n1`; both engine mailboxes receive it as a
   `ritual/offering` event.
2. The generated CLIPS rule `edge-e1-auto-pipe` matches the Offering's
   coire-event and fires the goal `caws_auto_pipe_e1('<cid>')`; the relayed
   goal runs `caws_pipe/4`, which re-publishes the cached Offering payload
   as an addressed Offering to `n2` (`consults/e1`, fresh correlation id).
   The CycleController registers a pending offer — convergence blocks.
3. `n2`'s RitualParticipant (addressed match) evaluates via Groq and
   replies Hohi echoing the correlation id and topic path.
4. `ingest_tephra` resolves the pending offer and writes the reply into
   both mailboxes; the generated `edge-e1-on-hohi-result` rule dispatches
   `caws_edge_reply('e1', hohi, '<cid>')`, which asserts
   `edge_result(e1, hohi, GroqReply)` (and would call `on_edge_hohi/2` if
   defined).
5. The root goal is re-proved: clause 1 of `reasoned_response/3` now finds
   the reply fact, ponders locally, synthesizes both answers, and the
   deduction converges. On Tabu or patience timeout the dispatch asserts
   `edge_result(e1, tabu, _)` instead and clause 2 answers with Clara
   alone — the Run degrades, never hangs.

## Gotchas (bit us during verification)

- (Fixed 2026-07-09, followup #2:) identical queries used to return the
  previous run's cached answer — the evaluate cache is now scoped per
  deduction, so re-running the same query re-evaluates.
- Dis's ritual registry survives restarts via CoireStore, but after a
  restart it is still safest to deactivate and re-activate the config.
- Focus `clara_mind_splinter` (the script does this) or the local ponder
  answers with the echo evaluator.
- lildaemon's pytest suite needs the server stopped (duckdb lock on
  `lildaemon.duc`).
